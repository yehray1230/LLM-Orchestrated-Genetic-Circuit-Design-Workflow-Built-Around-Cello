# 冷信寄送啟動路線圖

## 目標

在不等待完整技術升級路線圖全部完成的前提下，負責任地準備並寄出第一小批研究交流冷信。

寄信門檻並不是「專案已經完成」，而是：

> 相關領域的研究者能夠理解這個專案想解決的問題、查看一個可重現的案例、看懂專案的證據與宣稱邊界，並清楚知道你希望得到什麼回饋。

## 範圍鎖定

本路線圖只處理寄信前的準備工作。在第一批信件寄出以前，除非是為了讓旗艦案例能夠穩定重現，否則不要新增生物模型、Agent 角色、CRISPR 功能、科學資料來源、外部工具整合或額外案例。

準備期間出現的新想法應放入 backlog，不得成為寄信前的阻擋事項。

## 階段 0：定義這次寄信實驗

時間限制：半天。

- [ ] 為第一批信件選定一種主要受眾。
- [ ] 選定一個主要請求：技術回饋、15 分鐘短談，或請對方推薦更合適的研究者。
- [ ] 寫下一句話，說明為什麼這類受眾與專案有關。
- [ ] 第一批限制為 5 位收件人。
- [ ] 在開始後續階段之前，先訂下寄信日期。

第一批建議受眾：

研究 synthetic biology CAD、genetic circuit design、Cello、SBOL、computational biology 或 AI-assisted scientific design 的研究者。

完成條件：已明確寫下受眾、請求、收件人數量與寄信日期。

## 階段 1：凍結一個旗艦案例

時間限制：1–2 個專注工作日。

只使用一個案例：

```text
Activate GFP only when input A is present and input B is absent.
```

必要證據：

- [ ] 自然語言設計需求。
- [ ] 結構化規格與假設。
- [ ] Truth table 或同等的預期行為表示。
- [ ] Cello-compatible Verilog。
- [ ] 明確標示 `cello_mode`、mapping status，以及 mock 與 external Cello 的邊界。
- [ ] 一項容易閱讀的模擬或評估結果。
- [ ] Benchmark／evaluation 摘要。
- [ ] Readiness 狀態與主要阻擋原因。
- [ ] 一項 Critic finding 或 repair trace，用來說明多 Agent 迴圈的價值。
- [ ] 限制與下一個驗證步驟。
- [ ] 可重現案例的精確指令或操作文件。

這個案例不必證明濕實驗有效性、不必完成完整質體設計，也不必展示專案的所有功能。

完成條件：其他人不需要作者現場解說，也能依照文件操作並理解結果。

## 階段 2：執行證據檢查

時間限制：半天。

- [ ] 執行旗艦案例相關的聚焦測試。
- [ ] 執行寄信材料實際使用到的 demo checklist 路徑。
- [ ] 確認公開連結與指令能從乾淨的起始狀態正常使用。
- [ ] 確認產出不含憑證、私人路徑、私人資料集或未公開筆記。
- [ ] 確認所有分數都被描述為既有規則下的 heuristic 或 screening result，而不是生物成功機率。
- [ ] 確認 mock Cello output 不會被誤認為 external Cello mapping。
- [ ] 記錄測試指令、日期、結果與已知警告。

完成條件：對外可見的宣稱均有凍結後的成果支援，而且 demo 路徑已通過聚焦檢查。

## 階段 3：製作最小寄信材料包

時間限制：1 個專注工作日。

只製作以下材料：

1. 一頁式專案簡介，內容包括：
   - 想解決的問題；
   - 採用的方法；
   - 旗艦案例；
   - 已實作的成果；
   - 尚未驗證的部分；
   - 希望取得的回饋。
2. 一段 60–90 秒的畫面錄影；若錄影會延誤寄信日期，改用 4–6 張附註解的截圖。
3. 一份簡短、可重複使用的信件範本。
4. 一份收件人追蹤表，記錄姓名、相關性、個人化切入點、寄信日期、回覆與追蹤狀態。

完成條件：收件人不必安裝 repository，也能判斷這個專案是否與自己相關。

## 階段 4：撰寫並檢查信件

時間限制：半天。

信件應包含：

- [ ] 明確而具體的主旨。
- [ ] 一句個人化內容，說明為什麼聯絡這位收件人。
- [ ] 兩句話的專案介紹。
- [ ] 旗艦案例的一項具體結果。
- [ ] 一項明確限制或 research preview 邊界。
- [ ] 一個 repository 或專案簡介連結。
- [ ] 一項簡單且容易回答的請求。
- [ ] 簡潔的寄件人自我介紹。

目標長度：120–180 個英文字；若使用中文，則以一分鐘內能讀完為原則。不要把完整 roadmap 或所有功能清單放進信件。

完成條件：信件能在一分鐘內讀完，而且只有一個 call to action。

## 階段 5：寄出第一批信件

時間限制：在階段 0 訂下的日期寄出。

- [ ] 寄給 5 位經過仔細挑選的收件人。
- [ ] 至少針對開頭與相關性說明進行個人化。
- [ ] 將每封寄信記錄在收件人追蹤表中。
- [ ] 不因為非阻擋性的新功能想法而延後寄信。

完成條件：5 封信全部寄出並完成記錄。

## 階段 6：追蹤並學習

時間限制：首次寄信後 7–10 天。

- [ ] 對尚未回覆的人寄送一次禮貌的 follow-up。
- [ ] 將回覆分類為：有興趣、提供有效回饋、推薦他人、婉拒或未回覆。
- [ ] 記錄重複出現的問題與容易造成誤解之處。
- [ ] 只有在同一問題反覆出現或確實阻礙理解時，才修改專案材料。
- [ ] 判斷第二批信件是否應沿用同一受眾與訊息。

不要把 5 位收件人的沉默視為對專案的最終評價。這只是一次小規模的訊息與受眾測試。

完成條件：根據第一批觀察，為第二批信件寫下一份簡短的「保留／修改／停止」決定。

## 強制寄信門檻

以下五項全部成立時，就可以寄出第一批信件：

- [ ] 有一個可重現且容易理解的旗艦案例。
- [ ] 公開宣稱邊界清楚明確。
- [ ] Repository 或一頁式簡介提供清楚的閱讀起點。
- [ ] 信件具有一種明確受眾與一項明確請求。
- [ ] 對外展示所使用的 demo 路徑已通過聚焦驗證。

如果這五項都成立，即使後續技術 roadmap 尚未完成，也應寄出信件。

## 不得阻擋第一批信件的事項

- 完成技術 roadmap 的所有階段。
- 完成全部三個 demo cases。
- 取得濕實驗驗證或校正後的 in-vivo prediction。
- Phase 7b 科學資料整合。
- 完整 Phase 10 publication package。
- CRISPR 支援。
- 通用 plugin architecture。
- 完美的 UI、Logo、網站或宣傳影片。
- 支援所有外部工具。

## 變更管制規則

在階段 0–5 新增任何任務以前，先回答：

1. 缺少這項工作，是否會讓旗艦案例的結果不實或無法重現？
2. 缺少這項工作，是否很可能讓收件人誤解專案的宣稱邊界？
3. 缺少這項工作，是否會讓收件人無法查看或討論這個專案？

只有至少一題回答「是」時，才能加入寄信前的任務。否則應放入寄信後 backlog。

## 建議七天時程

| 日期 | 應完成的成果 |
| --- | --- |
| 第 1 天 | 凍結受眾、請求、5 位收件人與寄信日期。 |
| 第 2–3 天 | 完成並重現旗艦案例。 |
| 第 4 天 | 執行證據檢查並凍結成果。 |
| 第 5 天 | 製作一頁式簡介與截圖／影片。 |
| 第 6 天 | 撰寫、個人化並檢查 5 封信。 |
| 第 7 天 | 寄出第一批信件。 |

如果技術問題導致旗艦案例無法重現，可以移動一次寄信日期，並記錄確切阻擋原因。不得因視覺美化或新功能工作而移動日期。

## 完成定義

以下事項全部完成後，本路線圖才算結束：

- 第一批 5 封信已經寄出；
- 收件人與寄送日期均已記錄；
- 已完成一次 follow-up；
- 已根據觀察寫下第二批信件的簡短決定。
