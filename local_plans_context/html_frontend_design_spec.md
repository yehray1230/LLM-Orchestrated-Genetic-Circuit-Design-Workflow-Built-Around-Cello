# HTML 前端完整化設計規格

> 文件狀態：設計基準（implementation-ready product specification）  
> 建立日期：2026-07-03  
> 適用範圍：以 FastAPI、Jinja、HTML、CSS、JavaScript 為主的 `web/` 前端  
> 目標：讓 HTML 前端完整承接目前 Streamlit 與既有 Web 介面的主要功能，同時維持一般使用者容易理解、每頁聚焦且資訊完整。

## 1. 文件目的

本文件是未來規劃、設計與實作 HTML 前端時的主要參考。實作者應可從本文件得知：

- 新前端服務的主要使用者與資訊密度。
- 導覽架構與每個頁面的核心問題。
- 各頁必須呈現的功能、資料、操作與狀態。
- BYOK、模型、Cello 與其他進階設定應放置的位置。
- 哪些資訊可以在不同頁面重複，以及重複的目的。
- 版本、草稿、錯誤復原、證據、背景工作等橫跨全站的必要能力。
- 建議的分期實作順序與完成判準。

本文件描述的是產品與資訊架構，不代表所有後端契約目前都已存在。實作前仍需比對實際 route、service、schema 與資料持久化能力。

## 2. 已確定的產品方向

### 2.1 主要使用者

預設介面面向第一次或偶爾使用系統的一般使用者；研究人員、開發者與需要完整診斷資訊的人使用「進階／研究模式」。

一般模式不是刪除功能，而是：

- 優先顯示結論、狀態、風險與下一步。
- 將原始 JSON、完整事件、搜尋節點、模型參數等內容收進進階區域。
- 使用白話解釋專有名詞。
- 讓主要任務可以沿著明確步驟完成。

進階／研究模式應補充：

- 完整評分拆解與演算法資訊。
- 原始資料、事件、錯誤與 JSON。
- 搜尋樹、決策路徑及 Critic 回饋。
- 模型、工具、參數、資料來源與版本。
- 研究比較、重現性與 artifact。

### 2.2 三條硬性原則

1. **一頁一個主要問題**：每頁必須能用一句話說明「本頁幫使用者回答什麼」。
2. **允許有目的的資訊重複**：同一狀態或分數可以出現在首頁、設計總覽與比較頁，只要它在該頁支援不同決策。
3. **功能與資訊不可因簡化而消失**：一般模式可收合細節，但進階模式或對應專頁必須保留完整資訊。

### 2.3 不採用的方向

- 不把 Streamlit 的大量 tabs 原封不動搬到同一 HTML 頁。
- 不把所有模型、API key、Cello 與工程設定塞在首頁。
- 不只追求視覺漂亮而犧牲錯誤、警告、證據來源或原始診斷資訊。
- 不讓一般設計結果、研究級證據與 wet-lab 支持看起來是同一成熟度。

## 3. 建議的整體導覽

主導覽控制在七個入口：

1. 首頁
2. 設計案
3. 執行
4. 研究與評估
5. 組裝
6. 匯入
7. 設定

進入單一設計案後，顯示設計案次導覽：

`總覽 → 候選 → 比較 → 模擬 → 最佳化 → 組裝 → 匯出 → 歷程`

每個次頁保留設計名稱、ID、版本、狀態、成熟度、證據等級及「返回總覽」。

## 4. 頁面規格

### 4.1 首頁／總覽

**核心問題：我現在可以做什麼，哪些事情需要注意？**

建議內容順序：

1. 建立新設計與匯入設計的主要入口。
2. AI 服務狀態／BYOK 快捷入口。
3. 最近設計案。
4. 進行中的背景工作。
5. 待人工確認與重要警告。
6. 最近完成的研究、Benchmark、組裝與輸出摘要。

首頁只放摘要與下一步，不承擔完整分析。

#### AI 服務狀態卡

顯示：

- 系統模型可用、使用系統預設、使用自己的 API，或尚未設定。
- Provider 與模型名稱；不得顯示 API key 原值。
- 「設定自己的 API」或「變更模型」按鈕。
- 無可用模型時，明確說明必須先完成設定。

狀態層級：

- 綠色：模型可用，可直接建立設計。
- 黃色：正在使用系統預設，可選擇設定自己的 API。
- 紅色：沒有可用模型，必須先設定。

若部署環境沒有共用的系統模型，BYOK 應成為首次啟動引導；設定完成後縮回狀態卡。

### 4.2 建立設計

**核心問題：如何把自然語言需求轉成可以執行的設計規格？**

採精靈式流程：

1. 描述設計目標。
2. 選擇宿主與環境。
3. 確認輸入、輸出與邏輯。
4. 設定限制、偏好與運算預算。
5. 選擇模型、Cello 與進階工具（一般使用者可沿用預設）。
6. 最終確認並開始執行。

需求澄清／PM 問答應整合在此流程。每一步都要自動儲存、允許返回修改，並顯示完成進度。

模型區域只顯示簡短摘要，例如：

`AI 模型：系統預設 · <model name>　[變更]`

「變更」提供三種選擇：

1. 使用系統預設（一般使用者推薦）。
2. 使用自己的 API。
3. 本機或相容端點（進階模式）。

### 4.3 設計案總覽

**核心問題：這個設計現在是什麼狀態，下一步是什麼？**

必須包含：

- 原始需求與結構化規格。
- 設計版本、成熟度與驗證狀態。
- 最佳候選摘要。
- 主要分數及限制因素。
- 模擬、序列、宿主、組裝與匯出 readiness。
- 警告、缺少資料、暫定資訊與 unsupported gaps。
- 目前可用的輸出。
- 建議的下一步與主要操作按鈕。

這是設計案樞紐頁；不在此展開全部圖表和原始資料。

### 4.4 候選方案

**核心問題：單一候選的邏輯、生物詮釋與工程內容是什麼？**

必須包含：

- 候選列表、排名、狀態及是否為目前最佳。
- 邏輯提案與邏輯表達式。
- Verilog。
- 邏輯閘／拓撲圖。
- 調控關係圖。
- DNA construct 與 transcriptional units。
- Parts 清單及 mapping 狀態。
- 評分拆解與主要限制因素。
- Cello 結果、fallback 與 claim notice。
- 將候選設為最佳或建立修訂版的操作。
- 原始候選資料的進階摺疊區。

### 4.5 候選比較

**核心問題：哪個候選更適合目前的目標，原因是什麼？**

必須包含：

- 選擇 2–4 個候選。
- 共同指標與相容性檢查。
- 分數、限制因素、拓撲、模擬、元件與組裝差異。
- 推薦結論及理由。
- 不可直接比較時的明確警告。
- 從比較結果選定候選或建立版本。

比較頁可以重複候選詳情，但資訊必須重新組織成決策導向。

### 4.6 執行監控

**核心問題：工作流程目前做到哪裡，是否需要我介入？**

必須包含：

- 工作流程階段與即時狀態。
- 事件時間軸。
- 運算預算、時間、模擬次數與目前最佳分數。
- 目前候選、拓撲、ODE 摘要與搜尋路徑摘要。
- 失敗原因及已保存的成果。
- 暫停、取消、重試、恢復。
- 需要人工回饋時的輸入區。
- 下一次更新或 polling 狀態。

現有 HTML monitor 已有狀態 polling、事件、result、terminal、拓撲、ODE 與搜尋樹基礎；擴充時優先沿用既有 service 契約與 monitor view model。

### 4.7 搜尋與決策歷程

**核心問題：系統為什麼走到目前結果？**

必須包含：

- 搜尋樹與目前路徑。
- 每個節點的提案、模式、狀態與分數。
- Critic 回饋及修正行動。
- 保留或淘汰原因。
- 分數變化與決策時間軸。
- 人工決策與回饋紀錄。
- 原始節點及事件資料。

一般模式呈現摘要；完整樹、JSON 與 debug 資訊放在進階模式。

### 4.8 模擬與分析中心

**核心問題：設計在不同模型與條件下呈現什麼行為？**

至少拆成以下專頁：

- ODE 模擬。
- 隨機／SSA 模擬。
- 參數配適。
- 敏感度分析。
- 分岔與 transfer function。
- 圖表與指標摘要。

每個分析頁採一致結構：

1. 本分析回答的問題。
2. 輸入條件與參數。
3. 執行或重新執行操作。
4. 主要圖表及數值結果。
5. 白話解釋。
6. 限制、資料來源、fallback 與可信度。
7. 原始輸出及下載。

不應在同一頁同時展示所有分析圖表。

### 4.9 評估與研究

**核心問題：結果是否可比較、可重現，具有什麼證據強度？**

必須包含：

- Research runs 與 Benchmark runs。
- 執行版本、資料集、評分器及指標版本。
- 同版本與跨版本比較。
- 重現性資訊與 run manifest。
- 報告及研究 artifacts。
- preview、computational screening、provisional、unsupported、wet-lab supported 的清楚區分。

研究評估與一般候選比較必須分開，避免把工程候選排名誤認為學術驗證。

### 4.10 序列、宿主與最佳化

**核心問題：如何將候選推進到更具體且較可行的工程版本？**

必須包含：

- 序列分析及發現的問題。
- 修正建議與最佳化前後差異。
- 宿主相容性及 host calibration 狀態。
- 各 readiness domain 的結果。
- 採用或拒絕建議的操作。
- 從目前版本建立最佳化修訂版。
- 明確區分計算建議與實驗支持。

### 4.11 組裝工作台

**核心問題：如何把設計整理成可檢查、可下載的組裝規劃？**

必須包含：

- Backbone 管理與選擇。
- Parts 與 construct。
- 組裝方法與 assembly plan。
- BOM、引子、序列及其他 artifacts。
- 組裝驗證、衝突、警告與 readiness。
- Deliverable 產生、版本、下載及產生失敗狀態。

### 4.12 匯入

**核心問題：如何安全地把外部設計帶入系統？**

必須包含：

- 引導式建立與檔案上傳。
- 支援格式與範例。
- 格式、schema 與內容驗證。
- 預覽、警告、欄位修正。
- 確認建立設計案。
- 匯入來源、時間與 provenance。

### 4.13 匯出與報告

**核心問題：目前有哪些可信且可攜的輸出？**

應支援或預留：

- JSON。
- Verilog。
- SBOL3。
- GenBank。
- BOM CSV。
- Plasmid GenBank。
- 研究報告與組裝 artifacts。
- 完整可攜式專案包。
- 可列印、可分享的設計摘要。

每個輸出必須顯示成功、失敗、警告、版本與是否 assembly-ready。分享或報告不得包含 API key、內部敏感路徑或其他秘密。

### 4.14 設定與系統狀態

**核心問題：系統使用哪些模型、工具與預設值，是否正常可用？**

將完整技術設定放在此處，而非首頁：

- AI 使用模式：系統預設／BYOK／相容端點。
- Provider、模型、API key 與 API Base URL。
- 連線測試及作用中的設定摘要。
- 儲存範圍與清除憑證。
- Cello command、狀態與 claim boundary。
- 預設宿主與運算預算。
- 外部工具、adapter、版本、fallback 及健康狀態。
- artifact 與資料儲存位置。
- API／service health。

API key 欄位必須遮蔽；後端不可把已儲存的 key 原值送回前端。

## 5. BYOK 的最終放置決策

BYOK 應「容易發現，但不搶走主要任務」。

- **首頁**：顯示 AI 服務狀態與快捷入口，不放完整表單。
- **建立設計**：顯示目前模型摘要並允許快速切換。
- **設定 → AI 與模型**：提供完整 BYOK、Provider、model、base URL、測試與清除功能。
- **首次啟動**：僅在無系統模型可用時，要求先完成 BYOK 或相容端點設定。

一般使用者不應因看到 BYOK 而誤以為一定要準備自己的 API key。

## 6. 全站共用能力

### 6.1 共用頁面框架

每頁應視情況提供：

- 固定主導覽與設計案次導覽。
- 麵包屑。
- 頁面標題及一句「本頁可以回答什麼」。
- 設計名稱、ID、版本與狀態。
- 成熟度及證據等級。
- 最近更新與最後儲存時間。
- 主要下一步按鈕。
- 警告與待辦摘要。
- 原始 JSON／debug 摺疊區。
- 清楚的空狀態、載入狀態與錯誤狀態。

### 6.2 設計版本與分支

每次人工修改、最佳化、匯入或重新執行都不應默默覆蓋舊結果。需要：

- 版本時間軸。
- 從任一版本建立修訂版。
- 版本差異比較。
- 標記目前正式／選定版本。
- 回復或重新採用舊版本。
- 顯示版本來源：人工、工作流程、最佳化或匯入。

### 6.3 自動儲存與草稿恢復

- 建立設計時自動儲存。
- 顯示最後儲存時間。
- 提供未完成草稿列表。
- 離開未儲存內容前提醒。
- 失敗後保留輸入及已產生結果。
- 支援從中斷步驟繼續。

### 6.4 統一警告與待辦中心

集中顯示：

- 缺少必要資料。
- 等待人工確認。
- 模擬不完整或達到 step limit。
- Cello mapping 失敗或 fallback。
- 證據不足。
- 匯出或組裝尚未就緒。
- 建議下一步。

首頁顯示跨設計摘要；設計案顯示該設計完整清單。

### 6.5 證據與可信度

重要結論旁應能查看「為什麼」：

- 使用的模型、工具及版本。
- 輸入資料與參數來源。
- 規則推論、模擬或外部資料來源。
- confidence、provisional 與 fallback 狀態。
- 是否只有 computational screening。
- 是否有實驗支持。
- 尚缺哪些證據。

### 6.6 執行成本與時間

執行前及執行中顯示：

- 預計工作階段與預估時間。
- 運算預算。
- 可能使用的外部服務。
- BYOK 可能產生的費用提醒。
- 已使用時間、token、模擬次數或其他可量化預算。
- 「快速預覽」與「完整分析」選項。

不得提供無可靠依據的精確成本承諾；估算應標示為估算。

### 6.7 錯誤復原中心

錯誤頁或錯誤卡必須回答：

- 發生什麼事。
- 哪些成果已保存。
- 可重試哪個階段。
- 是否需要修改設定。
- 是否可以跳過非必要工具或使用 fallback。
- 如何複製診斷資訊。
- 如何在進階模式查看原始錯誤。

### 6.8 全域搜尋與篩選

- 搜尋名稱、ID、宿主與功能。
- 依狀態、成熟度、日期、warning 或 readiness 篩選。
- 收藏／釘選。
- 最近瀏覽。

### 6.9 資料與刪除管理

- 顯示資料及 artifact 的位置、大小、版本與建立時間。
- 匯出完整專案包。
- 封存設計。
- 刪除前預覽影響範圍。
- 清理失敗或過期執行產物。
- 對刪除、覆蓋、重新執行使用明確確認。

### 6.10 背景工作與通知

- 全站可見的進行中工作狀態。
- 完成、失敗及待確認通知。
- 背景工作佇列與同時執行數量。
- 清楚告知是否可安全關閉頁面。
- 從通知返回對應設計、執行或待辦。

### 6.11 新手引導

- 範例設計。
- 第一次使用檢查表。
- 術語提示。
- 情境式說明，而不只是一頁長文件。
- 一般／進階模式切換。
- 依目前狀態推薦下一步。

### 6.12 可及性與語言

- 支援中英文或預留一致的國際化結構。
- 專有名詞翻譯保持一致。
- 不只用顏色傳達狀態。
- 鍵盤可操作。
- 圖表提供文字摘要。
- 字級、對比與留白可讀。
- 小螢幕至少能查看狀態、通知和確認待辦。

## 7. 資訊重複規則

允許重複的典型資訊：

| 資訊 | 可出現位置 | 各位置目的 |
| --- | --- | --- |
| 設計狀態 | 首頁、設計總覽、所有設計子頁 | 找到工作、理解整體、維持上下文 |
| AI 模型狀態 | 首頁、建立設計、設定 | 可發現性、執行前確認、完整管理 |
| 最佳候選 | 設計總覽、候選、比較、匯出 | 摘要、檢查、決策、確認輸出來源 |
| 警告 | 首頁、待辦中心、設計總覽、相關功能頁 | 跨設計提醒、集中處理、整體風險、局部修正 |
| 成熟度／證據 | 設計總覽、研究、匯出、報告 | 避免誤解、學術判讀、輸出前確認、對外溝通 |
| 執行狀態 | 全站頁首、執行列表、執行監控 | 被動通知、工作管理、深入診斷 |

重複時必須使用相同來源資料與一致名詞，不得在不同頁自行重算出互相衝突的狀態。

## 8. 介面密度規則

- 一般內容區避免超過一個主要決策與兩個次要操作。
- 詳細表格、圖表、原始資料優先移到專頁或進階摺疊區。
- 不依賴左右兩個狹窄欄位比較大量內容；候選比較應使用專頁。
- 每個頁面先顯示摘要，再顯示證據與細節。
- 長表格提供搜尋、篩選、分頁或虛擬捲動。
- 空狀態必須告訴使用者為何沒有資料及如何產生資料。
- 禁用操作必須說明原因及解鎖條件。

## 9. 安全與資料邊界

- API key、token 與秘密不得出現在 HTML、log、debug JSON、報告或下載包。
- 儲存憑證時應說明作用範圍與清除方式。
- 破壞性操作需要確認與影響預覽。
- 外部服務呼叫前顯示會傳送的資料類型。
- 報告與分享輸出預設移除內部路徑、秘密與不必要的原始輸入。
- 生物設計 claim 必須保持 provisional、computational screening、unsupported gap 與 wet-lab supported 的邊界。

## 10. 建議實作順序

### Phase 0：盤點與契約確認

- 建立 Streamlit 功能、現有 HTML route/template、API/service 與資料模型對照表。
- 為每項功能標示：已完整、部分存在、只有 Streamlit、缺少後端契約。
- 確認一般／進階模式的顯示規則。

### Phase 1：共用框架與一般使用者主流程

- 新主導覽、設計案次導覽與共用頁首。
- 首頁及 AI 狀態卡。
- 建立設計精靈與草稿儲存。
- 設計案總覽及清楚的下一步。

### Phase 2：候選與執行完整化

- 候選詳情與候選比較。
- 執行監控、背景工作與通知。
- 人工回饋、重試與恢復。
- 搜尋／決策歷程專頁。

### Phase 3：模擬、最佳化與證據

- 拆分 ODE、SSA、敏感度、配適與其他分析頁。
- 序列、宿主與最佳化工作台。
- 統一 evidence、confidence、warning 與 readiness 元件。

### Phase 4：組裝、匯入、匯出與報告

- 組裝工作台與 deliverables。
- 完整匯入 review 流程。
- 匯出中心、可攜式專案包與分享安全。

### Phase 5：長期使用能力

- 版本、分支、差異與回復。
- 統一待辦與錯誤復原中心。
- 搜尋、篩選、封存與資料管理。
- 國際化、可及性與完整響應式 QA。

## 11. 優先級判斷

若資源有限，以下能力不得因為不夠亮眼而被延後到最後：

1. 設計版本與修訂。
2. 自動儲存與錯誤復原。
3. 統一警告與待辦。
4. 證據與可信度。
5. 背景工作狀態。
6. 成本、預算與執行前確認。

它們決定 HTML 前端是展示介面，還是真正可以取代 Streamlit 的工作環境。

## 12. 頁面完成判準

每個頁面在宣告完成前，至少確認：

- 能用一句話說明本頁核心問題。
- 一般使用者知道下一步是什麼。
- 進階資訊仍可被找到。
- loading、empty、success、partial、failed、cancelled 等狀態有合理呈現。
- provisional、fallback、unsupported 與 evidence 狀態沒有被隱藏。
- 操作失敗不會遺失已輸入資料。
- 危險操作有確認及影響說明。
- API key 或敏感資訊不會出現在前端 payload 或輸出。
- 鍵盤、窄螢幕與基本可及性可用。
- 有 route/service/template 或端到端測試覆蓋主要流程。
- 與其他頁面重複的狀態來自一致資料來源。

## 13. 未來實作前仍需決定的問題

以下不是本次討論的阻礙，但實作前需逐項決定：

- 一般／進階模式是使用者偏好、URL 參數，還是依角色決定。
- 系統部署是否提供共用模型；這會決定首次啟動是否強制 BYOK。
- API key 儲存位置、加密方式與 session／project／user 作用範圍。
- 是否需要登入、多使用者、權限與分享連結。
- 草稿、通知與版本資料需新增哪些持久化模型。
- 背景工作是否支援真正的 pause，或只能 cancel／resume 新工作。
- 成本與時間估算能取得哪些可靠資料。
- 中英文預設語言及術語表。
- 哪些 Streamlit 診斷功能應在 HTML 完整重現，哪些可由統一 raw/debug view 取代。

## 14. 最終設計判斷

新版 HTML 的核心不是增加更多頁面，而是把現有能力重新組織成清楚的任務路徑：

`建立需求 → 產生候選 → 理解與比較 → 模擬及修正 → 評估證據 → 組裝與輸出`

一般使用者預設看到「現在是什麼、能不能信、下一步做什麼」；研究與進階使用者則能繼續深入查看完整模型、參數、搜尋歷程、原始資料與重現性資訊。只要所有功能都有明確歸屬、狀態不被簡化掉，而且頁面維持一頁一重點，HTML 前端就能逐步成為完整主介面，而不只是 Streamlit 的縮小版。
